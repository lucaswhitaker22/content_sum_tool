SYSC4101 / SYSC5105: Definitions - Part II

Exhaustive Testing

  • Exhaustive testing is testing using all possible inputs.
  • Most of the time, this is impossible!

Examples:

  • A program that computes the factorial function:
    • Exhaustive testing would involve running the program with all possible integer values (0, 1, 2, ..., 100, ...).
  • A compiler (e.g., javac):
    • Exhaustive testing would involve compiling every possible Java program.

Techniques to Reduce the Number of Inputs:

  • Testing criteria group input elements into equivalence classes.
  • One input is selected from each class to ensure test data adequacy.
  • Criteria are used to decide which test inputs to use.
  • Criteria are used to decide when to stop testing.

Test Data: Procedure to Select?

  • Test data selection depends on the test model:
    • The test model can represent the specification (functional testing) or the implementation (structural testing).
  • Test objectives (requirements) define the criteria for selecting test data.
  • Test cases must exercise all elements in the test model.

Older Terminology:

  • Black-box testing: Functional testing.
  • White-box testing: Structural testing.

Functional vs. Structural Testing

Functional Testing:

  • Checks conformance with the specification.
  • Scales up to different levels of granularity:
    • Works for functions, classes, packages/components, and systems.
  • Depends on the specification and the degree of detail.
  • Does not reveal how much of the system (code) is being tested.
    • What if the system performs unexpected or undesirable tasks?

Structural Testing:

  • Based on control and data flow criteria.
  • Provides confidence about how much of the system is being tested.
  • Does not scale up well.
    • Mostly applicable at unit and integration testing levels.
    • Does not work for packages/components or systems.
  • Cannot reveal missing functionalities.
    • What if part of the specification is not implemented?

Diagram:

  • The system is represented by a box.
  • The specification is represented by a striped box within the system box.
  • The implementation is represented by a dotted box within the system box.
  • Missing functionality cannot be revealed by white-box testing.
  • Unexpected functionality cannot be revealed by black-box testing.

Test Model Criterion

  • Given a criterion C for a model M:
    • The coverage ratio of a test set T is the proportion of elements in M defined by C that are covered by T.
    • A test set T is C-adequate when the coverage ratio achieves 100% for criterion C.

Example 1:

  • Model M: Control flow graph of a function.
  • Criterion C: "All the statements."
  • Test suite: Exercises 5 out of 8 statements (62.5% coverage).
  • Conclusion: The test suite is not adequate for the all-statements criterion.

Example 2:

  • Model M: Set of use case scenarios.
  • Criterion C: "All the scenarios."
  • Test suite: Exercises 12 out of 12 scenarios (100% coverage).

  • Conclusion: The test suite is adequate for the all-scenarios criterion.

  • A test criterion specifies a set of test requirements/objectives.

  • Test requirements must be satisfied to obtain an adequate test suite.

Issue:

  • Not all test requirements are feasible when applying a criterion to a test model.

Revised Notion of Adequacy:

  • The coverage ratio of a test set T is the proportion of feasible elements in M defined by C that are covered by T.
  • A test set T is C-adequate when the coverage ratio achieves 100% for criterion C.

Theoretical Hierarchy of Criteria

  • Subsumption relation: C1 subsumes C2 if any C1-adequate test set is also C2-adequate.
  • Beware: This is not a subset relation! The set of model elements to be exercised to satisfy C2 is not a subset of the set of model elements to be exercised to satisfy C1.

Example:

  • Model M: Finite state machines.
  • Criterion C1: All-paths.
  • Criterion C2: All-transitions.
  • Conclusion: All-paths subsumes all-transitions. Any all-paths adequate test suite necessarily exercises all the transitions.

Usually (but not always), if C1 subsumes C2:

  • Satisfying C1 tends to be more expensive than satisfying C2. (e.g., C1 tends to require more test cases than C2).
  • A C1-adequate test suite tends to detect more faults than a C2-adequate test suite.

Two Ways to Use Test Criteria

  • Generate test values/test cases to satisfy the criterion.

    • Criterion = selection criterion.
    • Need a tool (or human), a generator, that automatically generates values to satisfy the criterion.
    • Wish: Create a (software) generator?

  • Evaluate coverage achieved by externally generated test values/test cases.

    • Criterion = coverage criterion.
    • Need a tool (or human), a recognizer, that automatically decides whether a set of values satisfies a criterion.
    • Wish: Create a (software) recognizer?

Problems:

  • How to create a generator?
  • How to create a recognizer?

Issue:

  • Both problems are provably undecidable for most criteria.
    • It is not possible to construct a single algorithm that will always, in every situation, find a correct solution.
  • However, it is often easier to build a recognizer than a generator.
    • Coverage analysis tools (recognizers) are quite plentiful.

Miss-Use of Terminology

  • Beware of miss-use of the term "coverage."
  • Your colleague says:
    • "I am checking what my tests exercise with the all-statements selection criterion."
      • Wrong: They are using the all-statements criterion in a recognizer context (coverage criterion).
    • "I am creating tests with the all-scenarios coverage criterion."
      • Wrong: They are using the all-scenarios criterion in a generator context (selection criterion).
    • "I am doing structural testing since I check my tests execute all statements."
      • Wrong: With structural testing, one uses a selection criterion that applies on a model of the implementation (generator context). Here, they are using the criterion in a recognizer context.
    • "My tests achieve 100% coverage."
      • What criterion? Different criteria may have extremely different costs!

Using a Test Selection Criterion

Steps:

  1. Choose a test model.
  2. Select a test criterion.
  3. Identify test objectives.
  4. Create test case specifications.
  5. Identify test data/input.
  6. Identify Oracle.

Example:

  1. Test model: State machine.
  2. Test criterion: All-transitions.
  3. Test objectives:
    • Got0 -> Got5
    • Got10 -> Got0
    • Got5 -> Got0
    • Got10 -> Got5
    • Got0 -> Got10
    • Got5 -> Got10
  4. Test case specifications:
    • TCS1 = [Got0, Got5, Got10]
    • TCS2 = [Got0, Got10, Got0]
    • TCS3 = [Got0, Got10, Got5, Got0]
  5. Test data:
    • TC1 = [5p, 5p]
    • TC2 = [10p, 5p]
    • TC3 = [10p, 10p, 10p]
  6. Oracles:
    • TC1 = machine has 10p in, no can
    • TC2 = machine has 0p in, one can
    • TC3 = machine has 0p in, two cans

Example (Graph) Model

  • Specification: A vending machine that gets 5p or 10p of money and sells 15p cans.
  • Test model: A graph representing the state transitions of the vending machine.
  • Test criterion: All-transitions.
  • Test objectives: All possible state transitions.
  • Test case specifications: Sets of transitions to be exercised.
  • Test data: Input values (5p or 10p) for each transition.
  • Oracles: Expected state after each transition.

Marick's Recommendation

Brian Marick recommends the following approach:

  1. Generate functional tests from requirements and design to try every function.
    • Use a functional selection criterion (generator).
  2. Check the structural coverage after the functional tests are all verified to be successful.
    • Use a structural coverage criterion (recognizer).
  3. Where the structural coverage is imperfect, generate functional tests (not structural) that induce the additional coverage.

This works because form (structure) should follow function!

  • Uncovered code must have some purpose, and that purpose has not been invoked, so some function is untested.

Test Criteria Based on Structure (Offutt)

  • Graphs:
    • Method body
    • Methods and calls
    • Components interactions
    • State and transitions
  • Logical Expressions:
    • Can appear in state machines, source code, and software specifications.
    • Examples: (not X or not Y) and A and B
  • Input Domain Characterization:
    • Describes the input domain of the software under test (method, component, system).
    • Examples:
      • A: {0, 1, >1}
      • B: {600, 700, 800}
      • C: {swe, cs, isa, infs}
  • Syntactic Structures:
    • Based on a grammar or other syntactic definition.
    • Example: Mutation testing

Test Criteria - Graphs

  • Example: Control flow graph of a function.
  • Criterion: All-transitions.
  • Test data: Input values that trigger each transition.
  • Oracles: Expected state after each transition.

Test Criteria - Logic Expressions

  • Example: A logical expression representing a condition in a program.
  • Criterion: All possible combinations of truth values for the variables in the expression.
  • Test data: Input values that result in each combination of truth values.
  • Oracles: Expected output of the expression for each combination of truth values.

Test Criteria - Input Domain Characterization

  • Example: A program that reads input files and searches for lines matching a regular expression.
  • Criterion: All possible input files and regular expressions.
  • Test data: Input files and regular expressions that cover the entire input domain.
  • Oracles: Expected output of the program for each input file and regular expression.

Test Criteria - Syntactic Structures

  • Example: A program with a specific syntax.
  • Criterion: All possible syntactic structures.
  • Test data: Input values that result in each syntactic structure.
  • Oracles: Expected output of the program for each syntactic structure.

Note: These are just examples of test criteria. There are many other criteria that can be used, depending on the specific software being tested.

Equivalence Class Testing
A technique used to reduce the number of inputs (i.e., test cases) by grouping input elements into equivalence classes, selecting one input from each class, and using criteria to determine which test inputs to use and when to stop testing.
Functional Testing
A type of testing that focuses on verifying the functionality of a system based on its specification.
Structural Testing
A type of testing that focuses on verifying the internal structure of a system, such as its code or control flow.
Test Criterion
A test criterion that specifies a set of test requirements or objectives that must be satisfied to obtain an adequate test suite.
Coverage Ratio
The proportion of elements in a model defined by a criterion that are covered by a test set.
C-Adequate
A test set that achieves 100% coverage for a given criterion.
Selection Criterion
A test criterion that is used to generate test values or test cases to satisfy a specific requirement.
Coverage Criterion
A test criterion that is used to evaluate the coverage achieved by externally generated test values or test cases.
Generator
A tool that automatically generates test values or test cases to satisfy a specific criterion.
Recognizer
A tool that automatically decides whether a set of values satisfies a specific criterion.
Subsumption
A relationship between two criteria for the same model where any test set that is adequate for the first criterion is also adequate for the second criterion.
Structural Test Criterion
A type of test criterion that focuses on the structure of a system, such as its control flow graph or state machine.
Input Domain Characterization
A type of test criterion that focuses on the input domain of a system, such as the range of values that can be accepted by a function or method.
Syntactic Structures
A type of test criterion that focuses on the syntactic structure of a system, such as its code or grammar.
Model-Based Testing
A type of testing that involves creating a model of the system under test and then using that model to generate test cases.

Q: What is exhaustive testing, and why is it often impractical?

A: Exhaustive testing is a testing technique that aims to cover all possible inputs to a program. However, it is often impossible to achieve due to the vast number of potential inputs, especially for complex programs.

Q: Explain the concept of equivalence classes in testing.

A: Equivalence classes are groups of inputs that are expected to produce similar outcomes. By selecting one representative input from each class, we can reduce the number of test cases while still achieving a good level of test coverage.

Q: What are the key differences between functional testing and structural testing?

A: Functional testing focuses on verifying that the software meets its specified requirements, without considering the internal implementation details. Structural testing, on the other hand, examines the internal structure of the code to ensure that all components are adequately tested.

Q: What is the purpose of a test model criterion?

A: A test model criterion defines a set of requirements or objectives that must be met for a test suite to be considered adequate. It helps to ensure that the test suite covers all critical aspects of the software.

Q: Describe the subsumption relation between test criteria, and provide an example.

A: The subsumption relation between test criteria indicates that if a test suite satisfies a more comprehensive criterion (C1), it automatically satisfies a less comprehensive criterion (C2). This means that C1 subsumes C2.

Q: Explain the roles of a test generator and a test recognizer in the context of test criteria.

A: A test generator is a tool or process that automatically creates test cases based on a given criterion. A test recognizer, on the other hand, determines whether a set of test cases satisfies a specific criterion.

Q: Summarize Brian Marick's recommendation for using test criteria.

A: Marick's recommendation emphasizes a two-step approach: first, generate functional tests to cover all functionalities, and then use structural coverage criteria to ensure that the internal code is adequately tested. This approach prioritizes functionality and uses structural testing to identify any potential gaps in coverage.

Short Answer Questions

Q: Explain the concept of exhaustive testing and how test criteria help to address its limitations.

A: Exhaustive testing aims to test all possible inputs to a system, but it is often impractical or impossible due to the vast number of potential inputs. Test criteria help to reduce the number of test cases by grouping inputs into equivalence classes, selecting representative inputs from each class, and defining rules for stopping testing when sufficient coverage is achieved.

Q: What is a test oracle and what is its role in software testing?

A: A test oracle is a mechanism used to determine whether a test case has passed or failed. It can be a human expert, a specification document, or a separate program that compares the actual output of the system to the expected output. The oracle plays a crucial role in evaluating the effectiveness of test cases and identifying defects.

Q: Explain the concept of subsumption relation between test criteria and provide an example.

A: The subsumption relation between test criteria indicates that one criterion is more comprehensive than another. If a test suite satisfies a more comprehensive criterion, it automatically satisfies the less comprehensive criterion. For example, the all-paths criterion subsumes the all-transitions criterion because any test suite that covers all possible paths in a state machine will also cover all possible transitions.

Long Answer Questions

Q: Explain the difference between functional and structural testing, highlighting their respective perspectives and goals.

A: Functional testing focuses on verifying that the software meets its specified requirements, while structural testing examines the internal structure of the code to ensure it is implemented correctly and efficiently. Functional testing is typically performed from a black-box perspective, where the tester does not have access to the source code, while structural testing is performed from a white-box perspective, where the tester has access to the source code. Both types of testing are essential for ensuring the quality of software, but they focus on different aspects of the system.

Q: What are test criteria and why are they important in software testing?

A: Test criteria are essential for defining the scope and objectives of testing. They provide a set of rules or guidelines that determine which aspects of the software should be tested and how thoroughly. By using test criteria, testers can ensure that their tests are comprehensive and effective in identifying defects. Some common test criteria include statement coverage, branch coverage, and path coverage, which aim to ensure that all parts of the code are executed at least once.

Multiple Choice Questions

Q: Which of the following test criteria is the most comprehensive and demanding in terms of test effort? a) Statement coverage b) Branch coverage c) Condition coverage d) All-paths coverage

  • a
  • b
  • c
  • d

Correct Answer: d

Explanation: The all-paths criterion is the most comprehensive, as it requires testing all possible execution paths through the software. This can be very time-consuming and expensive, but it provides the highest level of assurance that all parts of the code have been tested.

Q: Which of the following statements is TRUE about 100% statement coverage? a) It guarantees that all possible execution paths have been tested. b) It is the most comprehensive test criterion. c) It ensures that every line of code is executed at least once. d) It is always the most efficient way to test software.

  • a
  • b
  • c
  • d

Correct Answer: c

Explanation: A test suite that achieves 100% statement coverage ensures that every line of code is executed at least once. However, it does not guarantee that all possible branches or conditions within the code have been tested.

Q: What is the purpose of a test generator? a) To evaluate the coverage achieved by externally generated test values. b) To automatically create test cases based on a given test criterion. c) To identify potential defects in the software. d) To analyze the results of test execution.

  • a
  • b
  • c
  • d

Correct Answer: b

Explanation: A test generator is a tool that automatically creates test cases based on a given test criterion. It helps testers to generate a large number of test cases efficiently, which can be particularly useful for complex systems.

Q: Which of the following test criteria is specifically designed for testing state machines? a) All-transitions b) All-paths c) Statement coverage d) Branch coverage

  • a
  • b
  • c
  • d

Correct Answer: a

Explanation: The all-transitions criterion requires testing all possible transitions between states in a state machine. This ensures that all possible state changes are covered by the test suite.

Q: Which type of testing is more likely to uncover defects related to the internal structure of the code? a) Functional testing b) Black-box testing c) Regression testing d) Structural testing

  • a
  • b
  • c
  • d

Correct Answer: d

Explanation: Structural testing is based on the internal structure of the code, while functional testing focuses on verifying the behavior of the software against its specifications. Therefore, structural testing is more likely to reveal defects related to the implementation of the code, such as logic errors or incorrect data flow.
